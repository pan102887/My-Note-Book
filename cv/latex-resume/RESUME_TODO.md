# 简历问题与待办清单

本文档用于投递前自检。原则：简历中的每一个数字、技术方案和“熟练”都应能在面试中用真实案例解释，不确定的内容先核实再保留。

## P0：投递前必须完成

- [ ] 统一订单项目的性能指标口径。
  - 个人优势写的是“核心接口平均响应时间由 3 秒降低至 800 毫秒”，约降低 73%。
  - 项目经历写的是“订单处理时间减少 40%”。
  - 项目经历还写有“部分场景耗时缩短至原来的 1%”。
  - 明确这三个数据是否指不同接口、不同流程或不同统计口径；无法准确解释时删除“原来的 1%”。
- [ ] 为“并发处理能力提升 60%”补充定义：是峰值 QPS、TPS、消费速度还是压测吞吐量；记录优化前后数值、机器配置、数据量和压测条件。
- [ ] 核实“查询效率提升 30% 以上”的具体案例：SQL、表规模、索引变化、`EXPLAIN` 结果、优化前后耗时及统计口径。
- [ ] 核实 Redis 项目中的结果：请求成功率基线、用户投诉统计周期、流量规模，以及“未再发生重启或宕机”的观察周期。
- [ ] 确认 Kafka 项目中确实实施过“消息可靠投递、消费幂等”。若只是了解而未落地，应降低表述强度；若实施过，应准备具体机制。
- [ ] 明确订单一致性方案。仅使用 Kafka 或事件驱动不能自然保证一致性，需要说明实际使用的事务消息、本地消息表、幂等、重试、补偿或对账机制。
- [ ] 确认工作年限与当前时间一致。当前写法为“5 年”，应随投递时间更新。
- [ ] 确认年龄、邮箱、手机号、求职城市和在职状态均为最新信息。

## P1：显著提升通过率

### 工作经历

- [ ] 将工作经历从“负责什么”改成“负责范围 + 技术动作 + 业务结果”。
- [ ] 补充能够公开的业务规模：日均订单量、峰值 QPS、用户量、消息量、数据量或服务数量。
- [ ] 明确个人角色：独立负责、核心开发、方案设计、技术评审、上线负责人或稳定性保障。
- [ ] 增加一条重大活动保障经历，说明压测、容量评估、监控告警、降级预案和最终结果。
- [ ] 增加一个 2025 年之后的代表性成果，避免最近项目经历停留在 2025 年初。
- [ ] 弱化“官网、小程序日常维护”这类区分度低的描述，换成更具体的技术或业务成果。

### Kafka 订单异步化项目

- [ ] 说明哪些同步操作被异步化，哪些操作必须保留在核心链路中，以及这样划分的原因。
- [ ] 明确消息模型：Topic、事件类型、生产者、消费者、分区键和消费组。
- [ ] 补充同一订单消息有序性的处理方式，例如是否使用订单 ID 作为分区键。
- [ ] 补充生产端可靠性配置：`acks`、重试、发送回调、批量、压缩等实际使用项。
- [ ] 补充消费端可靠性：手动提交、幂等键、失败重试、死信、补偿、积压处理等实际使用项。
- [ ] 准备说明为何选择 Kafka，而不是 RocketMQ、RabbitMQ、线程池或同步调用。
- [ ] 明确性能测试方法，避免只给结果、不说明测试条件。

### Redis 令牌桶限流 SDK

- [ ] 明确限流维度：接口、用户、租户、活动或全局限流。
- [ ] 明确令牌生成和扣减是否通过 Lua 脚本保证原子性；若未使用 Lua，需要解释并发正确性。
- [ ] 说明 SDK 的接入方式：注解、拦截器、过滤器、AOP、配置中心或代码 API。
- [ ] 说明 Redis 不可用时采用 fail-open 还是 fail-closed，以及选择依据。
- [ ] 准备热点 Key、时钟误差、规则动态更新、本地兜底和集群扩展问题。
- [ ] 区分限流、熔断、降级三个概念，避免把“快速失败”一概描述为熔断。

### JVM、并发与数据库

- [ ] 准备一个 CPU 高占用排查案例：告警现象、排查命令、线程定位、代码根因、修复及验证。
- [ ] 准备一个 OOM 排查案例：OOM 类型、堆转储、分析工具、对象引用链、根因及修复。
- [ ] 准备一个慢 SQL 优化案例，能够现场解释索引结构、最左匹配、回表和执行计划。
- [ ] 准备一个分布式锁实际案例，说明锁超时、误释放、续期、主从切换和业务幂等问题。
- [ ] 对简历中使用“熟悉”的技术至少准备一个实际案例；对“能够独立完成”的能力准备完整项目证据。

## P2：版本化与呈现优化

- [ ] 针对不同 JD 维护至少两个版本：订单/交易方向、平台/基础架构方向。
- [ ] 根据 JD 调整个人优势和项目顺序，让最匹配的经历出现在上半页。
- [ ] 控制个人优势在 5～6 条，避免恢复为技术名词堆叠。
- [ ] 保持正文不小于当前字号，避免为了增加内容继续压缩页边距和行距。
- [ ] 导出 PDF 后检查文字是否可复制、邮箱链接是否有效、是否存在空白页或内容裁切。
- [ ] 使用招聘网站上传一次，确认 ATS 能正确解析姓名、工作经历、项目经历和教育经历。
- [ ] 文件名包含姓名、岗位和可选年份，例如 `潘少峰-Java后端开发-2026.pdf`。
- [ ] 投递外部岗位前考虑是否保留年龄、性别和头像，并按招聘市场习惯准备无头像版本。

## 每段项目经历的完成标准

一段项目经历应能回答以下问题后再投递：

- [ ] 为什么要做？业务痛点和技术瓶颈是什么？
- [ ] 我的职责边界是什么？哪些部分由我设计或实现？
- [ ] 系统规模是多少？
- [ ] 为什么选择这个方案？比较过哪些替代方案？
- [ ] 最难的问题是什么？如何定位和解决？
- [ ] 如何保障正确性、可靠性和可观测性？
- [ ] 结果如何测量？数据能否解释和复现？
- [ ] 如果重新设计，会改进什么？
