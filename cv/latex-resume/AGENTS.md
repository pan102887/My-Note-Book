# AGENTS.md

## 项目目标

本目录是潘少峰的中文 Java 后端求职简历工程。目标是维护一份内容真实、技术表述可信、便于 ATS 解析且可通过 XeLaTeX 稳定编译的 A4 PDF 简历。简历允许根据项目经历自然扩展为多页。

## 目录结构

- `resume.tex`：文档入口、页面布局、字体、颜色、个人基本信息及公共命令。
- `sections/summary.tex`：个人优势。
- `sections/experience.tex`：工作经历。
- `sections/projects.tex`：项目经历。
- `sections/education.tex`：教育经历。
- `assets/photo.png`：头像资源。
- `RESUME_TODO.md`：简历问题、数据核验和优化待办。
- `INTERVIEW_QUESTIONS.md`：根据简历整理的面试问题。
- `build/`：LaTeX 构建产物，不手工编辑。

## 构建与验证

在本目录执行：

```bash
make
```

预期输出：`build/resume.pdf`。

清理构建文件：

```bash
make clean
```

每次修改 `.tex` 文件后必须：

1. 使用 `make` 编译，确保命令返回成功。
2. 检查 `build/resume.log` 中是否存在 `Overfull`、`Underfull`、`Error` 或影响成品的警告。
3. 渲染或打开 `build/resume.pdf`，逐页检查是否存在裁切、重叠、意外空白页、孤立标题、异常分页或不可读文字。
4. 确认头像、日期、邮箱和量化指标没有因排版调整而丢失。

## 内容维护规则

- 不虚构业务规模、技术方案、职责或量化结果。
- 不擅自修改手机号、邮箱、年龄、工作时间等个人事实。
- 对不确定的信息使用 `TODO` 或在 `RESUME_TODO.md` 中记录，不能用猜测值补齐。
- 修改量化指标前，先核对指标对象、统计周期、压测条件及优化前后基线。
- “团队完成”与“个人负责”必须区分，避免把团队成果全部表述为个人成果。
- 使用“业务背景 → 个人职责 → 技术动作 → 结果”的方式描述项目，减少空泛的“负责、参与、熟悉”。
- 技术方案必须经得住追问。例如写“数据一致性”时，应能说明实际采用的幂等、重试、补偿、事务消息或本地消息表等机制。
- 避免绝对化承诺，如“完全不会丢失”“保证强一致”“再未发生任何故障”，除非有准确事实依据。
- 优先使用中文全角标点；技术名词、数字和单位前后保持当前文件中的空格风格。
- LaTeX 百分号必须写为 `\%`，特殊字符应正确转义。

## 排版规则

- 允许简历根据工作和项目经历自然扩展为多页，不以强行压缩到一页为目标。
- 新增内容时保持合理字号、行距和页边距，不通过牺牲可读性控制页数。
- 分页时避免章节标题单独落在页尾；同一公司或项目条目应尽量完整地排在同一页，无法避免跨页时应保证上下文连续。
- 后续页面应保持与首页一致的左右边距、正文宽度、章节层级和视觉风格。
- 正文建议保持约 10pt，不得因塞入内容显著降低可读性。
- 保持章节顺序：个人优势、工作经历、项目经历、教育经历。
- 保持条目标题、职位和日期的对齐方式一致。
- 不引入仅用于装饰、可能影响 ATS 解析的复杂图形、进度条或多栏技能图。
- 头像可通过 `\showphototrue` / `\showphotofalse` 切换；不要删除该能力。
- 默认使用 XeLaTeX，不切换到 pdfLaTeX，以免中文字体失效。

## 简历内容的重点风险

- 订单项目中的“3 秒降至 800 毫秒”“处理时间减少 40%”“并发能力提升 60%”及“缩短至原来的 1%”需要统一口径。
- Kafka 项目的消息可靠性、消费幂等、顺序性和数据一致性机制需要与真实实现相符。
- Redis 限流项目需明确令牌扣减原子性、限流维度、故障降级策略，以及限流与熔断的边界。
- “查询效率提升 30%”“请求成功率提升 30%”“投诉减少 50%”需要保留可解释的基线和统计方法。
- 当前项目时间集中在 2024～2025 年；后续应补充最近一年的代表性成果。
